# Промпт для тотального аудита движка Tectonica

Ты — геофизик-рецензент с опытом в вычислительной тектонике, геоморфологии и климатологии. Тебе дан исходный код процедурного генератора планет (`rust/planet_engine/src/lib.rs`, Rust → WASM, ~6500 строк). Движок генерирует тектонические плиты, рельеф, климат, биомы и гидрологию.

Твоя задача — провести полный научно-технический аудит в четыре этапа.

## Этап 1: Анализ кода

Прочитай `rust/planet_engine/src/lib.rs` целиком. Составь структурную карту:

1. **Pipeline** — порядок вызова всех стадий генерации (плиты → границы → деформация → кора → изостазия → рельеф → климат → биомы → гидрология). Для каждой стадии укажи:
   - Номера строк (от–до)
   - Входные данные
   - Выходные данные
   - Ключевые параметры с их значениями

2. **Вспомогательные функции** — все fn вне основного pipeline: шум, сглаживание, BFS, сортировка, утилиты. Для каждой: сигнатура, назначение, где используется.

3. **Структуры данных** — все struct: поля, размеры, lifetime.

4. **Числовые константы** — собери ВСЕ числовые литералы, magic numbers, пороги, коэффициенты. Для каждого:
   - Значение
   - Строка в коде
   - Физический смысл (если есть)
   - Источник (если указан в комментарии)

## Этап 2: Оценка научной модели

Для каждого физического процесса в коде оцени:

### 2.1 Тектоника
- Генерация плит (Вороной): насколько статистика форм/размеров соответствует Bird 2003?
- Классификация границ (conv/div/trans): корректность определения типа по относительному движению
- Деформация: damage rheology (Lyakhovsky 1997) — правильно ли реализована strain localization?
- Uplift: England & McKenzie 1982 crustal thickening — корректна ли формула? Правильные ли единицы?

### 2.2 Изостазия и литосфера
- Airy isostasy: корректность ρ_c, ρ_m, ρ_w, water loading
- Flexural isostasy: вывод N проходов из Te — математически верен?
- Thermal subsidence: GDH1 / Parsons & Sclater — правильная ли реализация?
- Isostatic relaxation: τ_eff = 5 Myr — обосновано?

### 2.3 Рельеф и геоморфология
- Stream power erosion (planet scope): параметры K, m, n
- SPACE erosion (crop scope): K_br, K_sed, H*, V_s, F_f — в допустимых диапазонах?
- Braun & Willett implicit scheme: корректность реализации, обновление приемника
- Diffusion: κ значение, количество проходов, Courant condition
- Вулканические дуги: Syracuse & Abers 2006 — d=166km, σ=40km — верно?
- Dynamic topography: Hager/Flament — амплитуды, длины волн
- Foreland basins: DeCelles & Giles — foredeep/forebulge расстояния и амплитуды
- Glacial buzzsaw: Brozovic — ELA formula, truncation logic
- Rift shoulders: Weissel & Karner — distance, amplitude
- Hotspots: Morgan/Crough — swell radius, amplitude
- Mid-ocean ridges: Macdonald — ridge/trench/fracture amplitudes
- Epeirogenic warping: Bond — amplitude range
- Back-arc basins: Karig/Sdrolias — distance behind arc, depth
- Cratonic peneplains: King — target elevation, activity threshold
- Oceanic plateaus: Coffin & Eldholm — radius, height ranges
- Continental shelf: Kennett — shelf break depth, width
- Sediment redistribution: Milliman & Syvitski — land fraction
- Coastline perturbation: Wessel & Smith — Gaussian weighting

### 2.4 Климат
- Температурная модель: полином широты, lapse rate, парниковый эффект
- Осадки: зональная модель, GPCP калибровка
- Ветер: трехклеточная циркуляция — корректность широтных зон
- Rain shadow: коэффициент, e-folding distance
- Windward moisture: L=700km — обосновано?
- Climate-dependent erosion (Hadley cell κ): Roe 2003 — правильные множители?

### 2.5 Гидрология
- Flow direction: D8 vs MFD — какой используется и где
- Flow accumulation: алгоритм, stochastic noise
- Озера: алгоритм заполнения
- Реки: порог площади водосбора

### 2.6 Биомы
- Whittaker classification: корректность полигонов
- Alpine zone: высотная зональность
- Биомный фильтр: физически ли обоснован?

Для каждого процесса дай оценку:
- **A** — научно корректно, параметры в опубликованном диапазоне
- **B** — качественно верно, параметры приблизительные но разумные
- **C** — упрощение с потерей важной физики
- **D** — эвристика без физического обоснования
- **F** — ошибка (неправильная формула, неверные единицы, баг)

## Этап 3: Вычисление хаков, эвристик и трюков

Пройди код строка за строкой. Найди ВСЕ случаи следующих категорий:

### 3.1 Fudge factors
Числовые константы, подобранные визуально/эмпирически без научного источника.
Формат: `[строка] значение — что делает — есть ли обоснование`

### 3.2 Эвристики
Алгоритмические приемы, заменяющие физику упрощенной логикой.
Формат: `[строки от-до] — описание — какую физику заменяет — насколько критично`

### 3.3 Хитрые математические трюки
Места где используются нестандартные приемы:
- Степенные функции вместо физических моделей (`.powf(3.0)`, `.powf(0.7)`)
- Clamp/min/max как замена boundary conditions
- Сглаживание как замена решения PDE
- Произвольные seed-зависимые хеши для "разнообразия"
- Линейная интерполяция вместо нелинейной физики
- Пороговые условия (`if x < 0.1 { continue }`) — откуда порог?

### 3.4 Скрытые допущения
- Какие физические процессы НЕЯВНО предполагаются мгновенными?
- Где нарушена причинность (эффект применяется до причины)?
- Где пространственный масштаб не соответствует физике?
- Где порядок операций влияет на результат но не обоснован?

### 3.5 Потенциальные баги
- Граничные условия на полюсах и линии перемены дат
- Off-by-one в индексации
- Деление на ноль / NaN propagation
- Переполнение f32
- Race conditions в последовательных проходах

Для каждого найденного элемента укажи:
- **Критичность**: 🔴 влияет на результат значимо / 🟡 влияет умеренно / ⚪ незначимо
- **Устранимость**: можно заменить физикой / неустранимо на данной архитектуре / требует рефакторинга
- **Рекомендация**: конкретное действие (формула, источник, параметр)

## Важно

- НЕ оценивай код по стандартам production software. Это научный генератор.
- Оценивай ФИЗИКУ, не code style.
- Если комментарий цитирует источник — проверь что формула в коде соответствует источнику.
- Если параметр вне опубликованного диапазона — это 🔴, даже если "работает визуально".
- Пустые комментарии типа "empirically tuned" без числового диапазона из литературы = fudge factor.
- Трюк "smooth N passes" вместо решения уравнения — всегда отмечай, даже если результат приемлем.
